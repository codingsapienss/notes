Utility Types + Generics

Now we are ready to understand one of the most powerful features of TypeScript:

Utility Types

Utility Types are built-in generic types provided by TypeScript that help us:

  • Reuse types
  • Transform types
  • Avoid duplication
  • Build scalable applications

Most utility types are implemented internally using:

Generics
+ keyof
+ Mapped Types

Utility Types are heavily used in:

  • React
  • Angular
  • NestJS
  • Express
  • APIs
  • Enterprise Applications

What Are Utility Types?

Utility Types are:

Pre-built Generic Types

provided by TypeScript.

Instead of writing:

New Types
Again and Again

we can transform existing types.

Example

type User = {
  id: number;

  name: string;

  age: number;
};

Using Utility Types we can create:

Optional User
Readonly User
Partial User
Selected User Fields
Excluded User Fields

without rewriting everything.

1. Partial

One of the most used utility types.

Problem

Suppose:

type User = {
  id: number;

  name: string;

  age: number;
};

Normally:

const user: User = {
  id: 1,
};

Error.

Because:

name missing
age missing

Partial Solution

type PartialUser = Partial<User>;

Generated Type

{
  id?: number;

  name?: string;

  age?: number;
}

All properties become:

Optional

Usage

const user: Partial<User> = {
  id: 1,
};

Valid.

How Partial Works Internally

Conceptually:

type Partial<T> = {
  [K in keyof T]?: T[K];
};

Meaning:

Loop through all keys
and make them optional

Real World Example

Update API

Original

type User = {
  id: number;

  name: string;

  age: number;
};

Update Request

function updateUser(userId: number, updates: Partial<User>) {}

Usage

updateUser(1, {
  name: "John",
});

Only changed fields required.

2. Required

Opposite of:

Partial<T>;

Example

type User = {
  id?: number;

  name?: string;
};

Transform

type FullUser = Required<User>;

Generated Type

{
  id: number;

  name: string;
}

All optional properties become:

Required

Usage

const user: Required<User> = {
  id: 1,

  name: "Prashant",
};

Missing fields:

❌ Error

Internal Idea

type Required<T> = {
  [K in keyof T]-?: T[K];
};

Meaning:

Remove optional marker (?)

Real World Example

After Database Save

Before Save

Partial<User>;

After Save

Required<User>;

because database guarantees all fields exist.

3. Pick<T, K>

One of the most important utility types.

Problem

Suppose:

type User = {
  id: number;

  name: string;

  age: number;

  email: string;
};

Need only:

id
name

Without Pick

type UserSummary = {
  id: number;

  name: string;
};

Duplicate code.

Pick Solution

type UserSummary = Pick<User, "id" | "name">;

Generated Type

{
  id: number;

  name: string;
}

Usage

const user: UserSummary = {
  id: 1,

  name: "Prashant",
};

Visualizing Pick

Original

User;
id
name
age
email

Pick

Pick<User, "id" | "name">;

↓

id
name

Real World Example

API Response

Database User

{
  id;
  name;
  email;
  password;
}

Frontend User

Pick<User, "id" | "name">;

Password never exposed.

4. Omit<T, K>

Opposite of Pick.

Problem

Need everything except:

password

Original

type User = {
  id: number;

  name: string;

  email: string;

  password: string;
};

Solution

type SafeUser = Omit<User, "password">;

Generated Type

{
  id: number;

  name: string;

  email: string;
}

Password removed.

Visualizing Omit

Original

id
name
email
password

Omit

Omit<User, "password">;

↓

id
name
email

Real World Example

Backend Response

Database Entity

User;

contains:

password
refreshToken

API Response

Omit<User, "password" | "refreshToken">;

Much safer.

Combining Utility Types

Extremely common.

Example

type User = {
  id: number;

  name: string;

  email: string;

  age: number;
};

Create

type UpdateUser = Partial<Pick<User, "name" | "email">>;

Generated Type

{
  name?:string;

  email?:string;
}

Very common in:

PATCH APIs
Update Forms

Utility Types + Generics

Suppose we create:

function updateEntity<T>(updates: Partial<T>) {}

Usage

updateEntity<User>({
  name: "John",
});

Type-safe.

Reusable.

Generic Repository Example

class Repository<T> {
  update(id: number, updates: Partial<T>) {}
}

Usage

repo.update(1, {
  name: "Prashant",
});

Only modified fields required.

API Example

interface ApiResponse<T> {
  success: boolean;

  data: T;
}

User Response

ApiResponse<Pick<User, "id" | "name">>;

Result

{
  success:true,

  data:{
    id:1,
    name:"Prashant"
  }
}

Utility Types Interview Pattern

Question:

Need all fields optional?

Answer:

Partial<T>;

Question:

Need all fields required?

Answer:

Required<T>;

Question:

Need only specific fields?

Answer:

Pick<T, K>;

Question:

Need everything except some fields?

Answer:

Omit<T, K>;

Common Mistakes

Creating Duplicate Types​

Bad

type UserSummary = {
  id: number;
  name: string;
};

Better

Pick<User, "id" | "name">;

Using any​

Bad

data: any;

Prefer

ApiResponse<T>;

Rewriting Update Types​

Bad

type UpdateUser = {
  name?: string;
  age?: number;
};

Better

Partial<User>;

Interview Questions

Q1​

What is Partial?

Answer​

Makes all properties optional.

Q2​

What is Required?

Answer​

Makes all properties required.

Q3​

What is Pick<T,K>?

Answer​

Selects specific properties.

Q4​

What is Omit<T,K>?

Answer​

Removes specific properties.

Q5​

Why are Utility Types important?

Answer​

They reduce duplication
and improve reusability.

Cheat Sheet

Partial<T>;

↓

All Optional
Required<T>;

↓

All Required
Pick<T, K>;

↓

Select Fields
Omit<T, K>;

↓

Remove Fields
Partial<Pick<User, "name">>;
Omit<User, "password">;

Key Takeaways

  • Utility Types are built-in generic helpers.
  • Partial<T> makes all properties optional.
  • Required<T> makes all properties required.
  • Pick<T,K> selects specific properties.
  • Omit<T,K> removes specific properties.
  • Utility Types are built using Generics and keyof.
  • They eliminate duplicate type definitions.
  • Utility Types are heavily used in APIs, forms, repositories and enterprise applications.
  • Combining Utility Types creates powerful reusable type transformations.
  • Mastering Utility Types is essential for professional TypeScript development.
