Access Modifiers

One of the biggest advantages of Classes is the ability to control:

Who can access what?

In real-world applications, not every property should be publicly accessible.

For example:

Bank Account Balance
User Password
Internal System IDs
Authentication Tokens

We need a mechanism to control visibility.

TypeScript provides:

  • public
  • private
  • protected
  • #private fields
  • readonly

These are called:

Access Modifiers

Why Access Modifiers?​

Consider:

class BankAccount {
  balance = 1000;
}

Create object:

const account = new BankAccount();

Anyone can do:

account.balance = -500000;

Problem:

Invalid State
Business Rules Broken
Data Corruption

We need:

Controlled Access

public Modifier​

What is public?​

Accessible Everywhere

This is the default modifier in TypeScript.

Example​

class User {
  public name: string;

  constructor(name: string) {
    this.name = name;
  }
}

Usage

const user = new User("Prashant");

console.log(user.name);

Output

Prashant

Modify

user.name = "John";

Valid.

public is Default​

These are identical:

class User {
  name: string;
}
class User {
  public name: string;
}

Most developers omit:

public;

because it is implied.

public Access Diagram​

Class
 │
 ├── Inside Class     ✅
 ├── Outside Class    ✅
 └── Child Class      ✅

private Modifier​

What is private?​

Accessible Only
Inside The Same Class

Outside access is prohibited.

Example​

class User {
  private password: string;

  constructor(password: string) {
    this.password = password;
  }
}

Create object:

const user = new User("123456");

Attempt

console.log(user.password);

Error.

Property 'password'
is private

private Access Diagram​

Class
 │
 ├── Inside Class     ✅
 ├── Outside Class    ❌
 └── Child Class      ❌

Accessing private Inside Class​

Private members can still be used internally.

Example

class User {
  private password: string;

  constructor(password: string) {
    this.password = password;
  }

  showPassword() {
    console.log(this.password);
  }
}

Valid because:

Same Class

Real World Example​

class BankAccount {
  private balance: number;

  constructor(balance: number) {
    this.balance = balance;
  }

  deposit(amount: number) {
    this.balance += amount;
  }

  getBalance() {
    return this.balance;
  }
}

Usage

const account = new BankAccount(1000);

account.deposit(500);

console.log(account.getBalance());

Output

1500

Attempt

account.balance = 0;

Error.

Why Use private?​

Protects:

Internal State
Business Rules
Sensitive Data

Examples:

Password
JWT Token
Account Balance
Database Connection

protected Modifier​

What is protected?​

Accessible:
Inside Class
+
Inside Child Classes

But NOT outside.

Example​

class Animal {
  protected name: string;

  constructor(name: string) {
    this.name = name;
  }
}

Child Class

class Dog extends Animal {
  bark() {
    console.log(this.name);
  }
}

Valid.

Because:

Child Class Can Access
protected Members

Outside Access​

const dog = new Dog("Tommy");

console.log(dog.name);

Error.

protected Access Diagram​

Class
 │
 ├── Inside Class     ✅
 ├── Child Class      ✅
 └── Outside Class    ❌

private vs protected​

private​

private balance: number;

Accessible:

Current Class Only

protected​

protected balance: number;

Accessible:

Current Class
+
Derived Classes

Example Comparison​

class Parent {
  private a = 10;

  protected b = 20;
}
class Child extends Parent {
  test() {
    console.log(this.b);

    console.log(this.a);
  }
}

Result

b → Valid

a → Error

# Private Fields (JavaScript Private Fields)​

Modern JavaScript introduced:

#property;

These are different from TypeScript private.

Example​

class User {
  #password: string;

  constructor(password: string) {
    this.#password = password;
  }
}

Attempt

user.#password;

Error.

Why Was # Introduced?​

TypeScript's:

private;

is only:

Compile-Time Protection

Generated JavaScript may still contain:

this.password;

But:

#password;

provides:

True JavaScript Runtime Privacy

TypeScript private vs #private​

TypeScript private​

private password: string;

Protection:

Compile Time

JavaScript #private​

#password: string;

Protection:

Runtime + Compile Time

readonly Modifier​

What is readonly?​

Allows:

Read Access

but prevents:

Modification

after initialization.

Example​

class User {
  readonly id: number;

  constructor(id: number) {
    this.id = id;
  }
}

Usage

const user = new User(101);

Read

console.log(user.id);

Valid.

Modify

user.id = 200;

Error.

Cannot assign to readonly property

Common Use Cases​

Examples:

Database IDs
Created At Timestamps
UUIDs
Order IDs
Employee IDs

These should never change.

readonly with Constructor Shorthand​

Example

class User {
  constructor(
    readonly id: number,
    public name: string,
  ) {}
}

Create

const user = new User(1, "Prashant");

Attempt

user.id = 5;

Error.

Combining Modifiers​

Very common.

Example

class User {
  public name: string;

  private password: string;

  protected role: string;

  readonly id: number;

  constructor(name: string, password: string, role: string, id: number) {
    this.name = name;
    this.password = password;
    this.role = role;
    this.id = id;
  }
}

Access Summary Table​

ModifierSame ClassChild ClassOutside Class
public✅✅✅
protected✅✅❌
private✅❌❌
#private✅❌❌
readonly✅ Read✅ Read✅ Read

Constructor Shorthand with Access Modifiers​

Very common in production.

Instead of:

class User {
  name: string;

  constructor(name: string) {
    this.name = name;
  }
}

Write:

class User {
  constructor(public name: string) {}
}

Even multiple properties:

class User {
  constructor(
    public name: string,
    private password: string,
    readonly id: number,
  ) {}
}

TypeScript automatically creates properties.

Real World Example​

class Employee {
  constructor(
    readonly id: number,
    public name: string,
    protected department: string,
    private salary: number,
  ) {}

  getSalary() {
    return this.salary;
  }
}

Usage

const emp = new Employee(1, "Prashant", "Engineering", 100000);

Valid

emp.name;

Valid

emp.getSalary();

Invalid

emp.salary;

Invalid

emp.department;

Common Mistakes​

Using private When Child Class Needs Access​

Wrong

private balance;

when subclass needs it.

Use:

protected balance;

Using readonly for Mutable Data​

Wrong

readonly balance;

if balance changes.

Use readonly only for:

Immutable Data

Assuming private Exists at Runtime​

TypeScript:

private;

does not create runtime protection.

Use:

#private;

when true runtime privacy is required.

Interview Questions​

Q1​

What is the default access modifier in TypeScript?

Answer​

public

Q2​

Difference between private and protected?

Answer​

private:
Only current class.

protected:
Current class + child classes.

Q3​

Can readonly properties be modified?

Answer​

No.
Only initialization is allowed.

Q4​

Difference between private and #private?

Answer​

private:
Compile-time protection.

#private:
Runtime + compile-time protection.

Q5​

When should protected be used?

Answer​

When derived classes need access
but external code should not.

Cheat Sheet​

public name: string;
private password: string;
protected role: string;
readonly id: number;
#token: string;
constructor(
  public name: string,
  private password: string
) {}

Key Takeaways​

  • Access Modifiers control visibility of class members.
  • public is the default modifier.
  • private restricts access to the same class only.
  • protected allows access inside child classes.
  • readonly prevents modification after initialization.
  • #private provides true JavaScript runtime privacy.
  • Constructor shorthand can automatically create properties.
  • Access modifiers help enforce encapsulation and protect internal state.
  • Choosing the correct modifier improves maintainability and security.
  • Access control is a fundamental OOP concept used heavily in production TypeScript applications.
