Interface vs Type

One of the most common TypeScript interview questions is:

Interface vs Type Alias

Both can be used to describe object structures.

Example:

interface User {
  name: string;
  age: number;
}

and

type User = {
  name: string;
  age: number;
};

Both appear identical.

So:

Why does TypeScript provide both?
When should we use each?
What are the differences?

This chapter answers those questions.

First Important Truth​

For most basic object definitions:

interface User {
  name: string;
  age: number;
}

and

type User = {
  name: string;
  age: number;
};

are effectively identical.

Both provide:

Type Safety
Autocompletion
Error Checking
Object Validation

Similarities​

Both can describe:

Objects​

Interface

interface User {
  name: string;
}

Type

type User = {
  name: string;
};

Function Types​

Interface

interface Add {
  (a: number, b: number): number;
}

Type

type Add = (a: number, b: number) => number;

Arrays​

Type

type Users = User[];

Interface

interface Users extends Array<User> {}

Possible, but type aliases are usually cleaner.

Both Support Extension​

Interface

interface User {
  name: string;
}
interface Admin extends User {
  permissions: string[];
}

Type

type User = {
  name: string;
};
type Admin = User & {
  permissions: string[];
};

Both achieve similar results.

Major Difference #1​

Declaration Merging​

This is the biggest difference.

Interface Supports Merging​

Example

interface User {
  name: string;
}
interface User {
  age: number;
}

Result

interface User {
  name: string;
  age: number;
}

TypeScript automatically combines them.

Usage

const user: User = {
  name: "Prashant",
  age: 25,
};

Valid.

Type Alias Does NOT Support Merging​

Example

type User = {
  name: string;
};
type User = {
  age: number;
};

Error

Duplicate Identifier

Type aliases cannot be reopened.

Visual Summary​

Interface

interface User {
  name: string;
}

interface User {
  age: number;
}

↓

{
  name: string;
  age: number;
}

Type

type User = {
  name: string;
};

type User = {
  age: number;
};

↓

Compilation Error

Major Difference #2​

Type Aliases Can Represent More Than Objects​

Interfaces mainly describe:

Object Shapes

Type aliases can describe:

Objects
Primitives
Unions
Intersections
Tuples
Mapped Types
Conditional Types

Primitive Types​

Possible

type ID = string;

Not Possible

interface ID extends string {}

Union Types​

Possible

type Status = "success" | "error" | "loading";

Interface?

interface Status {}

Impossible.

Tuple Types​

Possible

type User = [string, number];

Interface?

Not practical.

Conditional Types​

Possible

type Result<T> = T extends string ? number : boolean;

Interfaces cannot do this.

Major Difference #3​

Extension Syntax​

Interfaces use:

extends

Example

interface User {
  name: string;
}
interface Admin extends User {
  permissions: string[];
}

Type aliases use:

&

(Intersection)

Example

type User = {
  name: string;
};
type Admin = User & {
  permissions: string[];
};

Interface Extension​

Produces:

{
  name: string;
  permissions: string[];
}

Type Intersection​

Produces:

{
  name: string;
  permissions: string[];
}

Result looks similar.

Implementation differs.

Major Difference #4​

Error Messages​

Interfaces often produce cleaner errors.

Example

interface User {
  name: string;
}

Error

Type X is not assignable to User

Easy to understand.

Type aliases involving:

Union Types
Intersection Types
Generics
Conditional Types

can produce longer errors.

Major Difference #5​

Library Development​

Large libraries prefer interfaces for public contracts.

Example

interface Window {
  myAppVersion: string;
}

Why?

Because users can extend it later.

Example

interface Window {
  analytics: AnalyticsService;
}

Declaration merging makes this possible.

Type aliases cannot do this.

Interface Extension vs Type Intersection​

Frequently asked interview question.

Interface

interface A {
  name: string;
}
interface B extends A {
  age: number;
}

Result

{
  name: string;
  age: number;
}

Type

type A = {
  name: string;
};
type B = A & {
  age: number;
};

Result

{
  name: string;
  age: number;
}

Looks identical.

But intersections can create conflicts.

Property Conflict Example​

Type A

type A = {
  id: string;
};

Type B

type B = {
  id: number;
};

Intersection

type Result = A & B;

Result

{
  id: never;
}

Impossible type.

Interfaces catch conflicts earlier.

Performance Considerations​

The TypeScript team has mentioned that:

Interfaces are generally easier
for the compiler to optimize.

Difference is usually negligible.

Do not choose based on performance.

Choose based on design.

Modern Industry Practice​

Most large codebases follow:

Use Interface For

Object Models
API Contracts
Class Contracts
Public Libraries
Extendable Structures

Use Type Alias For

Union Types
Intersection Types
Tuples
Mapped Types
Conditional Types
Utility Types

Real World Example​

User Model

interface User {
  id: number;
  name: string;
}

Request Status

type Status = "loading" | "success" | "error";

This is extremely common.

Recommended Rule​

A practical rule used by many teams:

If describing:

Object Shape

Prefer:

interface;

If describing:

Anything Else

Prefer:

type;

Simple and effective.

Common Mistakes​

Using Type Alias for Everything​

Possible.

But you lose:

Declaration Merging
Interface Extension Benefits

Using Interface for Unions​

Wrong

interface Status {}

for:

"success" | "error";

Use:

type;

instead.

Forgetting Declaration Merging​

Interfaces merge.

Types do not.

Interview Questions​

Q1​

What is the biggest difference between Interface and Type?

Answer​

Interfaces support Declaration Merging.

Type Aliases do not.

Q2​

Can Interfaces represent Union Types?

Answer​

No.

Use Type Aliases.

Q3​

Can Type Aliases represent Primitive Types?

Answer​

Yes.

type ID = string;

Q4​

What is Declaration Merging?

Answer​

Multiple interfaces with the same name automatically combine.

Q5​

When should you use Interface?

Answer​

For object structures and public contracts.

Q6​

When should you use Type Alias?

Answer​

For unions, intersections, tuples and advanced type manipulation.

Cheat Sheet​

interface User {
  name: string;
}
type Status = "success" | "error";
interface Admin extends User {}
type Admin = User & {
  permissions: string[];
};
interface User {
  name: string;
}

interface User {
  age: number;
}

↓

{
  name: string;
  age: number;
}

Key Takeaways​

  • Interfaces and Type Aliases are similar for basic object definitions.
  • Interfaces support Declaration Merging.
  • Type Aliases support unions, tuples, primitives and advanced type transformations.
  • Interfaces use extends; Type Aliases use intersections (&).
  • Interfaces are preferred for object contracts and public APIs.
  • Type Aliases are preferred for advanced type manipulation.
  • Declaration Merging is the most important interview difference.
  • Modern TypeScript projects use both extensively.
  • Neither is universally better; each solves different problems.
  • Understanding both is essential for professional TypeScript development.
