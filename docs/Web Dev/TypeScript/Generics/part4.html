Generic Constraints, extends, keyof & Default Generics

Generics are extremely powerful because they allow us to write reusable code.

However, there is a problem:

Generics are TOO Flexible

A generic type can become:

string
number
boolean
object
array
class
interface

Sometimes we need to tell TypeScript:

"I want flexibility,
but not unlimited flexibility."

This is where:

Generic Constraints

become useful.

Why Do We Need Constraints?​

Consider:

function getLength<T>(value: T) {
  return value.length;
}

Looks correct.

Usage

getLength("Hello");

Works.

getLength([1, 2, 3]);

Works.

But:

getLength(100);

Error.

Why?

Because TypeScript only knows:

T;

could be:

Anything

And not every type has:

length;

property.

For example:

string   → has length
array    → has length
number   → no length
boolean  → no length

TypeScript Error

Property 'length'
does not exist on type 'T'

We need a way to say:

T must have a length property

Generic Constraints​

A Generic Constraint restricts what types can be used.

Syntax

<T extends SomeType>

Read it as:

T must satisfy SomeType

or

T must extend SomeType

First Constraint Example​

Create Interface

interface HasLength {
  length: number;
}

Generic Function

function getLength<T extends HasLength>(value: T) {
  return value.length;
}

Now TypeScript guarantees:

length exists

Valid

getLength("Hello");

Output

5

Valid

getLength([10, 20, 30]);

Output

3

Invalid

getLength(100);

Error.

Because:

number

does not satisfy:

HasLength;

Visualizing Constraints​

Without Constraint

<T>

means:

Anything

With Constraint

<T extends HasLength>

means:

Only types having length

Constraint Using Objects​

Interface

interface User {
  id: number;
}

Function

function printId<T extends User>(user: T) {
  console.log(user.id);
}

Valid

printId({
  id: 1,
  name: "Prashant",
});

Output

1

Invalid

printId({
  name: "Prashant",
});

Error.

Because:

id missing

Constraint With Multiple Properties​

Interface

interface Person {
  id: number;

  name: string;
}

Function

function display<T extends Person>(person: T) {
  console.log(person.id, person.name);
}

Valid

display({
  id: 1,
  name: "Prashant",
  age: 25,
});

Notice:

age;

is extra.

That is fine.

Constraint means:

Required properties must exist.
Extra properties are allowed.

Generic Constraints with Classes​

Class

class Animal {
  name = "Animal";
}

Function

function printName<T extends Animal>(animal: T) {
  console.log(animal.name);
}

Child Class

class Dog extends Animal {}

Usage

printName(new Dog());

Output

Animal

Because:

Dog;

satisfies:

Animal;

Understanding keyof​

One of the most important TypeScript concepts.

What is keyof?​

keyof extracts all property names from a type.

Example

type User = {
  id: number;

  name: string;

  age: number;
};

Using:

keyof User

produces:

"id" | "name" | "age";

Visualization

User
 ├── id
 ├── name
 └── age

↓

"id" | "name" | "age";

Why is keyof Useful?​

Suppose we want:

Safe Property Access

Without keyof

function getValue(obj: any, key: string) {
  return obj[key];
}

Problem

getValue(user, "salary");

Compiles.

But:

salary does not exist

Runtime bug.

Solution Using keyof​

function getValue<T, K extends keyof T>(obj: T, key: K) {
  return obj[key];
}

Let's understand this carefully.

T;

represents:

The Object Type
K;

represents:

The Property Name
keyof T

represents:

All keys inside T

Therefore:

K extends keyof T

means:

K must be a valid key of T

Example​

Object

const user = {
  id: 1,

  name: "Prashant",

  age: 25,
};

Valid

getValue(user, "name");

Output

Prashant

Valid

getValue(user, "age");

Output

25

Invalid

getValue(user, "salary");

Error.

Because:

salary

is not part of:

keyof User

Why keyof Is So Important​

Many built-in utility types use:

keyof;

internally.

Examples:

Pick;
Omit;
Record;
Partial;
Required;

Understanding:

keyof;

is essential before learning advanced TypeScript.

Repository Example Using Constraints​

Suppose we want a repository.

Without Constraints

class Repository<T> {
  private items: T[] = [];
}

Now imagine:

findById();

We need:

item.id;

But TypeScript doesn't know:

id exists

Solution​

class Repository<T extends { id: number }> {
  private items: T[] = [];

  add(item: T) {
    this.items.push(item);
  }

  findById(id: number) {
    return this.items.find((item) => item.id === id);
  }
}

Now:

TypeScript guarantees
id exists

Usage

const repo = new Repository<{
  id: number;
  name: string;
}>();

Valid.

Default Generic Types​

Sometimes we want:

A default type

when the developer doesn't provide one.

Syntax

<T = DefaultType>

Example​

class ApiResponse<T = string> {
  data!: T;
}

Usage

const response = new ApiResponse();

Type becomes:

ApiResponse<string>;

automatically.

Overriding Default Type​

Default

const response = new ApiResponse();

Type:

ApiResponse<string>;

Custom

const response = new ApiResponse<number>();

Type:

ApiResponse<number>;

Real World Example​

API Wrapper

interface ApiResponse<T = unknown> {
  success: boolean;

  data: T;
}

Usage

const userResponse: ApiResponse<User>;

Usage

const productResponse: ApiResponse<Product>;

If no type provided:

ApiResponse;

↓

ApiResponse<unknown>;

Very common in production code.

Combining Everything Together​

function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

Object

const user = {
  id: 1,

  name: "Prashant",
};

Usage

getProperty(user, "name");

Return Type

string;

Automatically inferred.

This is one of the most famous TypeScript Generic patterns.

Common Mistakes​

Using Unconstrained Generics​

Wrong

function fn<T>(value: T) {
  return value.length;
}

Use

<T extends HasLength>

Confusing extends with Inheritance​

In Generics:

<T extends User>

does NOT mean:

Inheritance only

It means:

Must satisfy User shape

Not Using keyof​

Wrong

key: string;

Better

K extends keyof T

Because:

Compile-time safety

Interview Questions​

Q1​

What is a Generic Constraint?

Answer​

A restriction placed
on a generic type.

Q2​

What does extends mean in Generics?

Answer​

The generic type
must satisfy another type.

Q3​

What is keyof?

Answer​

Extracts all property names
from a type.

Q4​

What does K extends keyof T mean?

Answer​

K must be a valid key of T.

Q5​

Why use Default Generics?

Answer​

To provide fallback types
when no generic is supplied.

Cheat Sheet​

interface HasLength {
  length: number;
}
keyof User

"id" | "name" | "age";

T[K];

(Property Type Lookup)

Key Takeaways​

  • Generic Constraints restrict generic types.
  • extends is used to create constraints.
  • Constraints improve type safety.
  • keyof extracts property names from a type.
  • K extends keyof T creates safe property access.
  • Constraints are heavily used in utility types and libraries.
  • Default Generics provide fallback types.
  • Repository and API patterns frequently rely on constraints.
  • keyof and constraints are foundational concepts for advanced TypeScript.
  • Mastering these concepts makes utility types and advanced generics much easier to understand.
