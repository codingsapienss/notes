Utility Types

As applications grow larger, we often need slightly modified versions of existing types.

Example:

User Type

but sometimes we need:

User Creation Type
User Update Type
User Public Profile Type
Readonly User Type

Creating separate types manually becomes repetitive.

TypeScript solves this using:

Utility Types

Utility Types are built-in generic types that help transform existing types into new types.

What are Utility Types?​

Utility Types are:

Built-in TypeScript helper types

used to create new types from existing types.

Instead of:

type User = {
  id: number;
  name: string;
  email: string;
};

and manually creating:

type UpdateUser = {
  id?: number;
  name?: string;
  email?: string;
};

TypeScript can generate it automatically.

Why Utility Types Exist?​

Without utility types:

type User = {
  id: number;
  name: string;
  email: string;
};

Now imagine:

CreateUser;
UpdateUser;
PublicUser;
ReadonlyUser;

You would constantly duplicate types.

Problems:

Code Duplication
Maintenance Issues
Inconsistency

Utility Types solve this.

Most Important Utility Types​

For interviews and real projects:

Partial<T>;
Required<T>;
Readonly<T>;
Pick<T, K>;
Omit<T, K>;

are the most important.

Base Type​

We'll use this type throughout the chapter.

type User = {
  id: number;
  name: string;
  email: string;
};

Partial​

Definition​

Makes:

All Properties Optional

Syntax​

Partial<Type>;

Example​

type User = {
  id: number;
  name: string;
  email: string;
};
type UpdateUser = Partial<User>;

Generated Type

{
  id?: number;
  name?: string;
  email?: string;
}

Usage​

const updateData: UpdateUser = {
  name: "Prashant",
};

Valid.

Only one property provided.

Why is Partial Useful?​

Common API Endpoint:

PATCH /users/1

User updates:

{
  name: "John";
}

Not entire object.

Partial perfectly models this.

Required​

Definition​

Makes:

All Properties Required

Example​

type User = {
  id?: number;
  name?: string;
};
type CompleteUser = Required<User>;

Generated Type

{
  id: number;
  name: string;
}

Usage​

Valid

const user: CompleteUser = {
  id: 1,
  name: "Prashant",
};

Invalid

const user: CompleteUser = {
  id: 1,
};

Error.

Why Use Required?​

Useful when:

Validation Completed

and data is guaranteed to exist.

Example:

DraftUser;

↓

ValidatedUser;

Readonly​

Definition​

Makes every property:

Read Only

Example​

type User = {
  id: number;
  name: string;
};
type ReadonlyUser = Readonly<User>;

Generated Type

{
  readonly id: number;
  readonly name: string;
}

Usage​

const user: ReadonlyUser = {
  id: 1,
  name: "Prashant",
};

Attempt:

user.name = "John";

Error.

Cannot assign to readonly property

Why Use Readonly?​

Protects objects from accidental modification.

Common examples:

Configuration Objects
Environment Variables
Application Settings
API Response Models

Pick<T, K>​

Definition​

Selects specific properties from a type.

Syntax​

Pick<Type, Keys>;

Example​

type User = {
  id: number;
  name: string;
  email: string;
};
type UserPreview = Pick<User, "id" | "name">;

Generated Type

{
  id: number;
  name: string;
}

Usage​

const preview: UserPreview = {
  id: 1,
  name: "Prashant",
};

Why Use Pick?​

Often we don't need the entire object.

Example:

User Card Component

needs:

id;
name;

only.

Not:

email;
address;
password;

Pick Multiple Properties​

type PublicUser = Pick<User, "id" | "name" | "email">;

Omit<T, K>​

Definition​

Removes properties from a type.

Syntax​

Omit<Type, Keys>;

Example​

type User = {
  id: number;
  name: string;
  email: string;
  password: string;
};
type PublicUser = Omit<User, "password">;

Generated Type

{
  id: number;
  name: string;
  email: string;
}

Why Use Omit?​

Very common in APIs.

Database Model

{
  id;
  name;
  email;
  password;
}

API Response

{
  id;
  name;
  email;
}

Never expose:

password

Pick vs Omit​

Pick​

Choose what you want.

Pick<User, "id" | "name">;

Omit​

Remove what you don't want.

Omit<User, "password">;

Real World Example​

Database Entity

type User = {
  id: number;
  name: string;
  email: string;
  password: string;
};

Create API Response

type PublicUser = Omit<User, "password">;

Result

{
  id: number;
  name: string;
  email: string;
}

Much safer.

Combining Utility Types​

Utility Types can be combined.

Example​

type User = {
  id: number;
  name: string;
  email: string;
};
type UpdateUser = Partial<Pick<User, "name" | "email">>;

Generated

{
  name?: string;
  email?: string;
}

Another Example​

type User = {
  id: number;
  name: string;
  email: string;
};
type ReadonlyPublicUser = Readonly<Omit<User, "email">>;

Generated

{
  readonly id: number;
  readonly name: string;
}

Base Type

Common Mistakes​

Confusing Pick and Omit​

Wrong mental model:

Pick removes properties

No.

Pick selects properties.

Assuming Readonly is Runtime Protection​

Wrong.

Readonly<User>;

works only during:

Compile Time

Generated JavaScript has:

No Readonly Enforcement

Overusing Partial​

Bad

Partial<User>;

everywhere.

Use it only when:

Data may be incomplete.

Interview Questions​

Q1​

What does Partial do?

Answer​

Makes all properties optional.

Q2​

What does Required do?

Answer​

Makes all properties required.

Q3​

Difference between Pick and Omit?

Answer​

Pick selects properties.

Omit removes properties.

Q4​

What does Readonly do?

Answer​

Makes all properties immutable.

Q5​

Can Utility Types be combined?

Answer​

Yes.

Example:

Readonly<Partial<User>>;

Cheat Sheet​

Partial<User>;

↓

{
  id?: number;
  name?: string;
}
Required<User>;

↓

{
  id: number;
  name: string;
}
Readonly<User>;

↓

{
  readonly id: number;
}
Pick<User, "id" | "name">;
Omit<User, "password">;

Key Takeaways​

  • Utility Types generate new types from existing types.
  • They reduce duplication and improve maintainability.
  • Partial<T> makes all properties optional.
  • Required<T> makes all properties mandatory.
  • Readonly<T> prevents property reassignment.
  • Pick<T, K> selects specific properties.
  • Omit<T, K> removes specific properties.
  • Utility Types can be combined together.
  • They are heavily used in APIs, React applications and backend systems.
  • Understanding these five utility types is essential for professional TypeScript development.
