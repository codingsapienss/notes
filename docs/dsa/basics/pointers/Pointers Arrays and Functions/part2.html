Character Arrays, Character Pointers, String Literals, and cout Behavior

Character arrays and character pointers are one of the most misunderstood areas in C++.

Most developers understand:

int *ptr;

but get confused when:

char *ptr;

behaves completely differently.

Questions that confuse most developers:

  • Why does cout << intPtr print an address?
  • Why does cout << charPtr print characters?
  • Why does cout << charArray print an entire string?
  • Why is char *ptr = "Hello" dangerous?
  • What happens if '\0' does not exist?
  • How does cout know where the string ends?

This chapter answers all of these questions.

Character Arrays Recap​

Consider:

char name[] = "Prashant";

Internally:

P r a s h a n t \0

Memory:

Address      Value

1000         P
1001         r
1002         a
1003         s
1004         h
1005         a
1006         n
1007         t
1008         \0

Important Observation​

Character arrays are stored:

One character per byte

because:

sizeof(char) == 1

Character Array Name​

Just like integer arrays:

char name[] = "Prashant";

Array name:

name

decays to:

&name[0]

Example​

char name[] = "Prashant";

cout << name << endl;

cout << &name[0];

Expected Output:

Prashant
Prashant

Wait... Why Not Addresses?​

For integer arrays:

int arr[5] = {1,2,3,4,5};

cout << arr;

Output:

0x7fff1234

Address.

But:

char name[] = "Prashant";

cout << name;

Output:

Prashant

Not an address.

Why?

The Most Important Concept​

cout has special handling for:

char*

and

const char*

Integer Pointer Example​

int value = 100;

int *ptr = &value;

cout << ptr;

Expected Output:

0x7fff1234

Address.

Character Pointer Example​

char name[] = "Prashant";

char *ptr = name;

cout << ptr;

Expected Output:

Prashant

Why Does This Happen?​

Because internally:

cout

contains special overloads.

Conceptually:

cout << int*

means:

Print Address

But:

cout << char*

means:

Treat pointer as string

Visual Representation​

Integer Pointer:

intPtr

↓

100

Output:

0x7fff1000

Character Pointer:

charPtr

↓

P r a s h a n t \0

Output:

Prashant

Excellent question.

char name[] = "Hello";

The Null Terminator​

Definition​

Null character:

'\0'

ASCII:

0

Purpose:

Marks end of C-style string

Why Is It Needed?​

Without:

'\0'

cout has no idea where the string ends.

char name[5] =
{

No '\0'

Undefined Behavior

'\0'

is accidentally found.

Very Common Interview Trap​

Consider:

char ch = 'A';

char *ptr = &ch;

cout << ptr;

Expected By Beginners​

Address

Actual Result​

Undefined Behavior.

Possible:

A▒▒▒▒...

Why?​

Memory:

Address      Value

1000         A
1001         ?
1002         ?
1003         ?

Problem​

cout assumes:

ptr

points to a string.

Execution​

Print A

Continue...

Continue...

Continue...

until:

'\0'

appears somewhere.

Correct Way To Print Address​

Use:

cout << static_cast<void*>(ptr);

Example​

char ch = 'A';

char *ptr = &ch;

cout << static_cast<void*>(ptr);

Expected Output:

0x7fff1234

Integer Array vs Character Array​

Integer Array​

int arr[5] =
{
1,2,3,4,5
};

cout << arr;

Output:

Address

Character Array​

char arr[] = "Hello";

cout << arr;

Output:

Hello

Why Different?​

Because:

cout

has special handling for:

char*

only.

Comparison Table​

TypeOutput
int*Address
double*Address
float*Address
char*String
const char*String

Character Array vs Character Pointer​

Many developers think these are identical.

They are related but different.

Character Array​

char name[] = "Hello";

Memory:

H e l l o \0

Stored directly.

Character Pointer​

char *ptr = name;

Memory:

ptr

↓

H e l l o \0

Difference​

Array:

Owns storage

Pointer:

Stores address

Example​

char name[] = "Hello";

char *ptr = name;

Size Comparison​

cout << sizeof(name);

Output:

6
cout << sizeof(ptr);

Output:

8

(64-bit system)

String Literals​

Consider:

"Hello"

Internal Representation​

H e l l o \0

Compiler automatically adds:

'\0'

String Literal Memory​

String literals are stored in:

Read-Only Memory

Dangerous Example​

char *ptr = "Hello";

This compiles on some compilers (or gives warnings).

Internal Memory​

ptr

↓

Read-only Memory

H e l l o \0

Why Dangerous?​

Consider:

ptr[0] = 'M';

Result​

Undefined Behavior

Possible:

Segmentation Fault

Correct Approach​

char name[] = "Hello";

name[0] = 'M';

Output:

Mello

Why Safe?​

Because:

char name[]

creates writable memory.

Behind The Scenes​

char name[] = "Hello";

Compiler does:

Temporary literal:

H e l l o \0

Then copies into:

Array memory

Visual​

Literal

↓

"Hello"

↓

Copy

↓

name[]

Modern C++ Recommendation​

Instead of:

char *ptr = "Hello";

use:

const char *ptr = "Hello";

because:

String literal is read-only.

Common Bugs​

Bug 1​

char ch='A';

char *ptr=&ch;

cout<<ptr;

Reason:

Not a string

Bug 2​

char name[5]={'H','e','l','l','o'};

cout<<name;

Reason:

Missing '\0'

Bug 3​

char *ptr="Hello";

ptr[0]='M';

Reason:

Read-only memory

Bug 4​

Thinking:

char*

always prints address.

Wrong.

Interview Questions​

Q1​

Why does:

cout << charPtr;

print characters?

Answer​

Because cout treats char* as a C-style string.

Q2​

Why does:

cout << intPtr;

print an address?

Answer​

No special string handling exists.

Q3​

What is the role of '\0'?

Answer​

Marks end of C-style string.

Q4​

What happens if '\0' is missing?

Answer​

Undefined behavior.

Q5​

Why is:

char *ptr = "Hello";

dangerous?

Answer​

String literal resides in read-only memory.

Q6​

How do you safely represent a string literal?

Answer​
const char *ptr = "Hello";

Cheat Sheet​

char name[] = "Hello";

char *ptr = name;

cout << ptr;

cout << static_cast<void*>(ptr);

char *ptr = &name[0];

char *ptr = name + 2;

const char *ptr = "Hello";

sizeof(name);

sizeof(ptr);

Key Takeaways​

  • Character pointers behave differently from other pointers.
  • cout treats char* as a C-style string.
  • '\0' marks the end of a string.
  • Missing '\0' causes undefined behavior.
  • Character arrays own storage.
  • Character pointers store addresses.
  • String literals are stored in read-only memory.
  • Never modify string literals through pointers.
  • Prefer const char* when pointing to string literals.
  • Understanding char* behavior is critical for interviews and legacy C/C++ code.
