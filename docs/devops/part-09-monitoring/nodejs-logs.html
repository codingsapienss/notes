Node.js Logs

Overview​

After Nginx receives a request and forwards it to your application, the Node.js application becomes responsible for handling the request.

While Nginx logs network-level information such as requests and HTTP status codes, Node.js logs provide visibility into what happens inside the application.

Application logs help developers and administrators answer questions like:

  • Did the API receive the request?
  • Which database query failed?
  • What exception occurred?
  • Why did the application crash?
  • Which user encountered an error?
  • How long did an operation take?

Without proper logging, troubleshooting production issues becomes extremely difficult.

Learning Objectives​

After completing this chapter, you will be able to:

  • Understand application logging.
  • Generate logs in Node.js.
  • Use console logging effectively.
  • Monitor PM2 logs.
  • Understand Morgan and Winston.
  • Create structured logs.
  • Handle application errors.
  • Apply production logging best practices.

What Are Application Logs?

Application logs are records generated by your application while it is running.

Unlike Nginx logs, which record HTTP requests reaching the server, application logs describe what happens after the request enters your code.

Example:

Application logs provide insight into the internal behavior of the application.

Basic Logging with console.log()

Node.js provides simple logging through the console object.

Example:

console.log("Application started");

Log variables.

console.log("User:", user.name);

Log objects.

console.log(user);

Although simple, console.log() is useful during development and for basic debugging.

Console Methods

Node.js provides multiple logging methods.

MethodPurpose
console.log()General information
console.info()Informational messages
console.warn()Warnings
console.error()Errors
console.table()Display tabular data
console.time()Start timer
console.timeEnd()Stop timer

Example:

console.error("Database connection failed");

Logging Errors

Never silently ignore errors.

Example:

try {
  // Database operation
} catch (err) {
  console.error(err);
}

Error logs should include:

  • Error message
  • Stack trace
  • Timestamp
  • Request details (when appropriate)

PM2 Logs

When Node.js applications are managed using PM2, all console output is automatically captured.

View application logs.

pm2 logs

View logs for a specific application.

pm2 logs app

Monitor logs in real time.

Application

↓

PM2

↓

Log Files

↓

Administrator

PM2 simplifies log collection for production deployments.

PM2 Log Files

PM2 stores logs by default in:

~/.pm2/logs/

Example:

app-out.log

app-error.log

These files separate normal output from errors.

Morgan

Morgan is an HTTP request logging middleware for Express.

Example installation:

npm install morgan

Basic usage:

const morgan = require("morgan");

app.use(morgan("dev"));

Morgan automatically logs:

  • Request method
  • URL
  • Status code
  • Response time
  • Response size

It is useful for monitoring incoming HTTP requests.

Winston

Winston is one of the most popular logging libraries for Node.js.

Installation:

npm install winston

Basic example:

const winston = require("winston");

const logger = winston.createLogger({
  transports: [new winston.transports.Console()],
});

logger.info("Application started");

Winston provides much greater flexibility than console.log().

Structured Logging

Production applications should generate structured logs.

Instead of:

User Login Failed

Prefer:

{
  "level": "error",
  "userId": 1024,
  "route": "/login",
  "message": "Invalid password",
  "timestamp": "2026-07-20T10:15:00Z"
}

Structured logs are easier to search, filter, and analyze.

Log Levels

Different messages should use different severity levels.

LevelPurpose
ErrorCritical failures
WarnPotential issues
InfoGeneral application events
DebugDevelopment information
TraceVery detailed debugging

Using appropriate log levels improves readability and filtering.

What Should Be Logged?

Important events include:

  • Application startup
  • Application shutdown
  • Database connections
  • User authentication
  • API requests
  • Validation failures
  • Exceptions
  • External API failures
  • Payment processing
  • Background jobs

Avoid excessive logging of unimportant events.

What Should Never Be Logged?

Never log sensitive information.

Examples:

  • Passwords
  • OTPs
  • Credit card numbers
  • JWT secrets
  • API keys
  • Database passwords
  • Session secrets
  • Encryption keys

Sensitive information in logs can become a serious security risk.

Log Rotation

Without log rotation, log files continue growing indefinitely.

Example:

↓

↓

Large log files consume disk space and slow log analysis.

PM2 supports log rotation through additional modules, and Linux systems commonly use tools such as logrotate.

Real-World Example

A production Express.js application begins returning 500 Internal Server Error responses.

The administrator investigates:

  1. Nginx Access Logs confirm requests are reaching the application.
  2. PM2 logs show repeated exceptions.
pm2 logs
  1. The application log contains:
TypeError:

Cannot read properties of undefined

  1. Developers trace the error to a missing request parameter.

  2. Input validation is added to prevent invalid requests.

  3. After redeployment, the application operates normally.

The application logs identified the exact line of code responsible for the failure.

Best Practices

  • Use meaningful log messages.
  • Include timestamps in every log entry.
  • Use log levels consistently.
  • Separate normal output from error logs.
  • Use structured logging for production applications.
  • Rotate logs regularly.
  • Avoid logging sensitive information.
  • Correlate application logs with Nginx and system logs.
  • Log exceptions with stack traces.
  • Monitor logs continuously in production.

Common Mistakes

Using Only console.log()​

While useful during development, console.log() alone is insufficient for large production applications.

Logging Sensitive Information​

Passwords, tokens, and API keys should never appear in logs.

Logging Everything​

Excessive logging makes it difficult to identify important events and increases storage usage.

Ignoring Error Logs​

Unhandled exceptions should always be investigated rather than repeatedly restarting the application.

Forgetting Log Rotation​

Unmanaged log files can consume significant disk space and eventually affect server stability.

Summary

Node.js logs provide visibility into the internal behavior of an application after requests reach the backend. By using console logging, PM2 logs, Morgan, Winston, structured logging, and appropriate log levels, administrators and developers can efficiently diagnose production issues and monitor application health. When combined with Nginx and system logs, application logs provide a complete end-to-end view of request processing and system behavior.
