# Static File Serving

## Overview

In the previous chapter, we learned how Nginx acts as a **Reverse Proxy**, forwarding dynamic requests to backend applications like Node.js.

However, not every request requires backend processing.

Consider these files:

- HTML
- CSS
- JavaScript
- Images
- Fonts
- Videos
- PDFs
- Downloads

These are **static files**.

Since static files do not change for every request, Nginx can serve them directly without involving the backend application.

Serving static files through Nginx significantly improves:

- Performance
- Scalability
- Resource utilization
- Response time

This is one of the primary reasons Nginx is placed in front of application servers.

---

## Learning Objectives

After completing this chapter, you will be able to:

- Understand what static files are.

- Understand request mapping.

# What Are Static Files?

Static files are files whose contents remain the same until they are manually modified.

Examples include:

- HTML pages
- CSS stylesheets
- JavaScript bundles
- Images
- Videos
- Fonts
- PDF documents
- ZIP archives

Unlike API responses or dynamically generated pages, static files are simply read from disk and returned to the client.

---

# Static vs Dynamic Content

| Static Content     | Dynamic Content      |
| ------------------ | -------------------- |
| CSS                | API Responses        |
| Images             | Database Queries     |
| JavaScript Bundles | Authentication       |
| Fonts              | User Dashboards      |
| PDFs               | Shopping Cart        |
| Videos             | Personalized Content |

Static content usually does not require application logic.

Dynamic content is generated by backend applications.

---

# Why Use Nginx for Static Files?

Suppose a browser requests:

```text
/logo.png
```

If the request goes through Node.js:

```text
Browser
     │
     ▼
Node.js
     │
     ▼
Read File
     │
     ▼
Browser
```

The application spends CPU time and memory handling a request that simply returns an existing file.

With Nginx:

```text
Browser
     │
     ▼
Nginx
     │
     ▼
Static File
     │
     ▼
Browser
```

The backend application is not involved.

---

# Performance Benefits

Serving static files through Nginx provides several advantages:

- Lower CPU usage on the application server
- Reduced memory consumption
- Faster response times
- Better handling of concurrent requests
- Reduced backend workload

This allows Node.js to focus on business logic instead of file delivery.

# Basic Static File Configuration

Example:

```nginx
server {

    listen 80;

    server_name example.com;

    root /var/www/html;

}
```

Directory:

```text
/var/www/html/

index.html

css/

images/

js/
```

Request:

```text
http://example.com/images/logo.png
```

Nginx automatically maps the URL to:

```text
/var/www/html/images/logo.png
```

---

# Understanding the root Directive

The `root` directive specifies the base directory for serving files.

Example:

```nginx
location / {

    root /var/www/html;

}
```

Mapping:

```text
Request URL

/images/logo.png

↓

Root Directory

/var/www/html

↓

Filesystem

/var/www/html/images/logo.png
```

Nginx appends the request path to the configured root directory.

# Example: Serving Downloads

Directory:

```text
/var/www/files/

manual.pdf

setup.zip
```

Configuration:

```nginx
location /downloads/ {

    root /var/www/files;

}
```

Request:

```text
/downloads/manual.pdf
```

File served:

```text
/var/www/files/downloads/manual.pdf
```

Notice that the request path is appended to the configured root directory.

# Understanding the alias Directive

Sometimes the URL path should not match the filesystem path.

This is where `alias` is useful.

Example:

```nginx
location /downloads/ {

    alias /var/www/files/;

}
```

Now:

```text
URL

/downloads/manual.pdf

↓

Alias

/var/www/files/

↓

Filesystem

/var/www/files/manual.pdf
```

Unlike `root`, the matched URL prefix is replaced by the alias path.

---

# root vs alias

Consider the same request:

```text
/downloads/report.pdf
```

Using `root`:

```nginx
location /downloads/ {

    root /var/www/files;

}
```

Result:

```text
/var/www/files/downloads/report.pdf
```

Using `alias`:

```nginx
location /downloads/ {

    alias /var/www/files/;

}
```

Result:

```text
/var/www/files/report.pdf
```

Comparison:

| Directive | Result                                                           |
| --------- | ---------------------------------------------------------------- |
| `root`    | Appends the request path to the configured directory             |
| `alias`   | Replaces the matched location path with the configured directory |

# Serving an Entire Website

Suppose the website files are:

```text
/var/www/company/

index.html

about.html

css/

images/

js/
```

Configuration:

```nginx
server {

    listen 80;

    server_name company.com;

    root /var/www/company;

    index index.html;

}
```

Visitors requesting:

```text
company.com
```

receive:

```text
/var/www/company/index.html
```

# Combining Static Files and Reverse Proxy

Most production applications use both.

```text
Internet
      │
      ▼
Nginx
      │
 ┌────┴───────────┐
 ▼                ▼
Static Files   Reverse Proxy
 │                │
 ▼                ▼
Filesystem     Node.js
```

Example:

| URL        | Handler |
| ---------- | ------- |
| `/css/`    | Nginx   |
| `/images/` | Nginx   |
| `/fonts/`  | Nginx   |
| `/api/`    | Node.js |
| `/login`   | Node.js |

This reduces unnecessary work for the backend.

# Real-World Example

Suppose an online learning platform stores:

- Course images
- PDF notes
- JavaScript bundles
- CSS files

The Express.js application handles:

- User authentication
- Payments
- Progress tracking
- APIs

Instead of forwarding every request to Node.js, Nginx serves:

```text
/images/

/css/

/js/

/downloads/
```

directly from the filesystem.

Only requests such as:

```text
/api/

/login

/profile
```

are forwarded to the backend application.

This architecture improves performance while reducing CPU and memory usage on the application server.

# Best Practices

- Let Nginx serve static assets whenever possible.
- Organize static files into dedicated directories.
- Use `root` when the URL structure matches the filesystem structure.
- Use `alias` when the URL should map to a different directory.
- Keep static assets separate from application code.
- Restrict access to private files that should not be publicly available.
- Test configurations using `nginx -t` before reloading.

---

# Common Mistakes

### Serving Static Files Through Node.js

This increases application workload and reduces scalability when Nginx can serve the files directly.

---

### Confusing root and alias

Using the wrong directive often results in incorrect filesystem paths and "404 Not Found" errors.

---

### Mixing Public and Private Files

Sensitive documents should never be placed in directories that Nginx serves publicly.

---

### Forgetting the Filesystem Structure

The configured directory must actually contain the files referenced by incoming URLs.

---

### Not Testing Configuration

Always validate configuration changes before reloading Nginx.

```bash
sudo nginx -t
```

---

# Summary

Static file serving is one of Nginx's greatest strengths. By serving HTML, CSS, JavaScript, images, fonts, videos, and downloadable files directly from the filesystem, Nginx reduces the workload on backend applications and delivers content efficiently. Understanding the `root` and `alias` directives allows administrators to map URLs to filesystem paths correctly and build fast, scalable production deployments where Nginx handles static content and backend applications focus on dynamic business logic.
