PM2

Overview​

Running a Node.js application using:

node app.js

works well for development, but it is not suitable for production.

If the application crashes, the process stops. If the server reboots, the application does not start automatically. There is also no built-in monitoring, log management, or process control.

This is where PM2 comes in.

PM2 is a production-grade process manager for Node.js applications. It keeps applications running, automatically restarts them after failures, starts them when the server boots, and provides monitoring and log management.

PM2 is one of the most widely used tools for deploying Node.js applications on Linux servers.

Learning Objectives​

After completing this chapter, you will be able to:

  • Understand what PM2 is.
  • Learn why process managers are required.
  • Install PM2.
  • Start and stop applications.
  • Monitor running applications.
  • Configure automatic startup.
  • View logs.
  • Follow production best practices.

  • SSH connection disconnects

The application becomes unavailable until someone manually starts it again.

This greatly improves application availability.

Installing PM2

PM2 is usually installed globally.

sudo npm install -g pm2

Verify installation:

pm2 --version

Example:

6.x.x

Starting an Application

Suppose your project contains:

my-app/
│
├── app.js
├── package.json
└── .env

Start the application:

pm2 start app.js

PM2 creates a managed process.

Naming Applications

Assign a meaningful name:

pm2 start app.js --name my-app

Example:

PM2

↓

my-app

Using descriptive names makes managing multiple applications easier.

Viewing Running Applications

Display all managed processes:

pm2 list

Example:

┌────┬─────────┬────────┬──────┐
│ id │ name    │ status │ cpu  │
├────┼─────────┼────────┼──────┤
│ 0  │ my-app  │ online │ 0%   │
└────┴─────────┴────────┴──────┘

This command provides a quick overview of running applications.

Stopping an Application

Stop an application:

pm2 stop my-app

Restart it:

pm2 restart my-app

Delete it from PM2:

pm2 delete my-app

Deleting removes the application from PM2's process list but does not delete the project files.

Monitoring Applications

PM2 provides built-in monitoring.

pm2 monit

Example:

Application

↓

CPU Usage

↓

Memory Usage

↓

Restart Count

↓

Event Loop Delay

This is useful for identifying resource issues during production.

Viewing Logs

PM2 automatically captures:

  • Standard output
  • Error output

View logs:

pm2 logs

Logs for a specific application:

pm2 logs my-app

Clear logs:

pm2 flush

Logs are invaluable when troubleshooting application problems.

Application Information

View detailed information:

pm2 show my-app

Typical information includes:

  • Process ID
  • Status
  • Uptime
  • Restart count
  • CPU usage
  • Memory usage
  • Script path
  • Environment variables

Saving the Process List

After configuring applications:

pm2 save

PM2 stores the current process list.

PM2 Process List

↓

Saved

This enables applications to be restored automatically after a reboot.

Starting PM2 on Boot

Generate the startup configuration:

pm2 startup

PM2 displays a command similar to:

sudo env PATH=$PATH:/usr/bin pm2 startup systemd -u ubuntu --hp /home/ubuntu

Run the generated command.

Finally:

pm2 save

Now:

Server Boot

↓

PM2 Starts

↓

Applications Start Automatically

Without these steps, applications will not restart after the server reboots.

Updating an Application

A common deployment workflow:

git pull
npm ci
pm2 restart my-app

Flow:

GitHub
    │
git pull
    │
npm ci
    │
PM2 Restart
    │
Updated Application

PM2 reloads the updated application without requiring manual process management.

Running Multiple Applications

PM2 can manage several applications simultaneously.

Example:

PM2
│
├── api-server
├── admin-panel
├── worker
└── scheduler

Each application can be started, stopped, monitored, and restarted independently.

PM2 and Environment Variables

PM2 automatically passes environment variables to Node.js applications.

Example:

NODE_ENV=production pm2 start app.js

or by using an ecosystem configuration file, which will be covered later.

The application accesses values through:

process.env;

PM2 Directory Structure

PM2 stores configuration and logs in the user's home directory.

Typical structure:

~/.pm2/
│
├── logs/
├── pids/
├── modules/
├── dump.pm2
└── pm2.log
Directory/FilePurpose
logs/Application logs
pids/Process ID files
dump.pm2Saved process list

Real-World Example

Suppose you deploy an Express.js application.

Project:

/var/www/my-app
│
├── app.js
├── package.json
├── .env
└── node_modules/

Deployment:

cd /var/www/my-app

git pull

npm ci

pm2 restart my-app

If the application crashes due to an unexpected error, PM2 automatically detects the failure and starts a new process. If the Linux server is rebooted after maintenance, PM2 restores the saved process list and starts the application automatically, minimizing downtime.

Best Practices

  • Install PM2 globally using npm.
  • Assign meaningful names to applications.
  • Enable PM2 startup after server boot.
  • Run pm2 save after configuration changes.
  • Monitor CPU and memory usage regularly.
  • Review application logs periodically.
  • Run applications as a dedicated non-root user.
  • Keep PM2 updated.

Common Mistakes

Running Applications with node app.js​

Applications started directly with Node.js stop if the terminal closes or the process crashes.

Forgetting pm2 save​

Without saving the process list, applications will not restart automatically after a reboot.

Ignoring Logs​

Many production issues can be diagnosed quickly by reviewing PM2 logs.

Running as Root​

Applications should generally run under a dedicated application user rather than the root account.

Managing Too Many Applications Without Names​

Using descriptive names makes administration significantly easier when multiple services are hosted on the same server.

Summary

PM2 is a production-ready process manager that keeps Node.js applications running reliably on Linux servers. It provides automatic restarts, startup on system boot, process monitoring, centralized logging, and simple lifecycle management. By using PM2 instead of running applications directly with node, administrators can significantly improve application reliability and simplify production deployments.
