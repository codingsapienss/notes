Fail2Ban

Overview​

Even after securing SSH with SSH keys and configuring a firewall, your server will continue receiving thousands of automated login attempts from bots across the Internet.

These bots continuously scan public IP addresses looking for:

  • Weak SSH passwords
  • Exposed services
  • Default usernames
  • Vulnerable applications

A firewall allows or blocks traffic based on predefined rules, but it does not detect suspicious behavior.

This is where Fail2Ban becomes valuable.

Fail2Ban continuously monitors system log files for repeated failed login attempts and automatically blocks the offending IP addresses for a specified period.

It acts as an Intrusion Prevention System (IPS) for Linux servers.

Learning Objectives​

After completing this chapter, you will be able to:

  • Understand brute-force attacks.
  • Learn how Fail2Ban works.
  • Understand jails and filters.
  • Configure Fail2Ban.
  • Protect SSH using Fail2Ban.
  • View banned IP addresses.
  • Apply production best practices.

Why Do We Need Fail2Ban?

Imagine exposing SSH to the Internet.

Within a few minutes, your logs may contain entries like:

Failed password for root

Failed password for ubuntu

Failed password for admin

Failed password for test

Failed password for user

These attempts are usually generated by automated bots.

Even if they never succeed, they:

  • Waste server resources.
  • Fill authentication logs.
  • Continuously probe your server.

Fail2Ban automatically detects this behavior and blocks the offending IP.

What is a Brute-Force Attack?

A brute-force attack repeatedly tries different passwords until one succeeds.

Attacker
     │
password123
     │
wrong
     │
admin123
     │
wrong
     │
qwerty
     │
wrong
     │
...

Without protection, attackers can continue indefinitely.

What is Fail2Ban?

Fail2Ban is an intrusion prevention tool that:

  • Monitors log files.
  • Detects repeated failed authentication attempts.
  • Temporarily bans offending IP addresses.
  • Updates firewall rules automatically.
Authentication Logs
        │
        ▼
Fail2Ban
        │
Detect Attack
        │
        ▼
Firewall
        │
Block IP

The attacker can no longer reach the protected service during the ban period.

How Fail2Ban Works

The workflow is straightforward.

Client
   │
Failed Login
   │
Authentication Log
   │
Fail2Ban Reads Log
   │
Too Many Failures?
   │
Yes
   │
Firewall Rule Added
   │
Client Blocked

Everything happens automatically.

Fail2Ban Components

Fail2Ban is organized into several components.

ComponentPurpose
JailDefines what to monitor and what action to take
FilterDefines the log patterns to detect attacks
ActionSpecifies what happens when an attack is detected
Log FileSource of authentication events

What is a Jail?

A jail combines:

  • A log file
  • A filter
  • A firewall action

Example:

SSH Jail

↓

Monitor auth.log

↓

Detect Failed Login

↓

Ban IP

Each service can have its own jail.

Examples include:

  • SSH
  • Nginx
  • Apache
  • FTP
  • Postfix

Installing Fail2Ban

On Ubuntu:

sudo apt update
sudo apt install fail2ban

Start the service:

sudo systemctl enable fail2ban
sudo systemctl start fail2ban

Verify status:

sudo systemctl status fail2ban

Configuration Files

Main configuration:

/etc/fail2ban/jail.conf

Recommended practice:

/etc/fail2ban/jail.local

Do not modify jail.conf directly.

Instead, create or edit jail.local so updates do not overwrite your configuration.

Example SSH Jail

A common configuration:

[sshd]
enabled = true
port = ssh
logpath = %(sshd_log)s
maxretry = 5
findtime = 10m
bantime = 1h

Explanation:

SettingMeaning
enabledEnables the jail
portProtected service
logpathLog file to monitor
maxretryFailed attempts allowed
findtimeTime window for counting failures
bantimeDuration of the ban

Understanding the Configuration

Suppose:

maxretry = 5

findtime = 10 minutes

bantime = 1 hour

This means:

If an IP address fails authentication five times within ten minutes, it will be blocked for one hour.

Checking Fail2Ban Status

Overall status:

sudo fail2ban-client status

Example:

Status

Number of Jails: 1

Check a specific jail:

sudo fail2ban-client status sshd

Example:

Currently Banned: 3

Banned IP List:

192.168.1.10
198.51.100.20
203.0.113.15

Unbanning an IP

Remove a banned address:

sudo fail2ban-client set sshd unbanip 203.0.113.15

Useful if a legitimate administrator is accidentally blocked.

Fail2Ban does not replace the firewall.

Instead, it dynamically updates firewall rules based on suspicious activity.

Best Practices

  • Use Fail2Ban together with SSH keys.
  • Keep Fail2Ban updated.
  • Configure reasonable maxretry and bantime values.
  • Monitor banned IP addresses periodically.
  • Protect additional Internet-facing services where appropriate.
  • Store custom settings in jail.local instead of modifying jail.conf.

Common Mistakes

Thinking Fail2Ban Replaces SSH Keys​

Fail2Ban is an additional security layer.

SSH key authentication should still be the primary authentication method.

Setting Extremely Long Ban Times​

Very long bans may accidentally block legitimate administrators.

Choose ban durations appropriate for your environment.

Editing jail.conf Directly​

Package updates may overwrite jail.conf.

Use jail.local for custom configurations.

Ignoring Authentication Logs​

Fail2Ban blocks attacks, but reviewing authentication logs remains important for detecting unusual activity and understanding attack patterns.

Summary

Fail2Ban is an intrusion prevention tool that strengthens Linux server security by monitoring log files for suspicious activity and automatically blocking IP addresses that repeatedly fail authentication. Working alongside UFW, SSH security, and cloud firewalls, it provides an additional layer of defense against brute-force attacks without requiring constant administrator intervention. In production environments, Fail2Ban is a lightweight yet highly effective component of a layered security strategy.
